# ソフトウェア開発テーブルトークRPG基本ルール
## はじめに
このゲームは,チームによるソフトウェア開発を疑似体験するのを目的としている。  
プレイヤーは,なんらかのソフトウェアを開発するチームの一員としてゲームをプレイする。  
プレイヤーは,作成した（もしくは与えられた）キャラクターの能力を活かして様々なタスクに取り組む。  
例えば,マネージャーとして参加しているプレイヤーはチームメンバーに仕事を割り振ったり,スケジュールの管理といったタスクに取り組む。  
チームメンバーとして参加しているプレイヤーは,モジュールの設計やプログラミングを行うといったタスクに取り組むことになる。


## 用語

| 用語 | 意味 |
|:----|:---|
| 2d6  |６面体サイコロを２つ振り合計を求めるダイスロールを行うことを指す |
| GM   | ゲームマスターの略,ゲームの司会進行役 |

## 基本方針,基本的な考え方について
 ### 数値の大小
  * 数値は大きい方がより価値が高いものとみなす
  * 難易度は数値が大きいほど難しい
 ### ルールの厳密性
 * ダイスロールの結果が物語に致命的な影響を及ぼしてしまった場合,なんらかの救済措置をとるのを推奨する
   - 結果をなかったコトにするかわりに何か別なペナルティを付与する
   - 少し過去に戻って結果を覆すような事前準備があったことにする（「こんな事もあろうかと・・・」）
   - なんでもいいから言い訳を考える(「ヘルメットが無ければ即死だった」)
   - ・・・という夢をみたことにする
 * 迷ったら物語がよりドラマチックになる方向で判断する
 * 最終決定はGMが下す
 * 物語が面白くなることが正義


## キャラクターの特徴
キャラクターはゲームルール上は３種類の要素によって特徴を表現する。
1. 能力値（アビリティ）  
 数値による,キャラクターの精神的特徴,他のRPGと異なり肉体的特徴は無い。肉体的な特徴は特性によって表現する。
1. スキル  
 そのキャラクターの身につけている技能を表現する。
1. 特性  
 そのキャラクターの個性や特徴,生活環境を表現する。

### 能力値
能力値はそれぞれ,2-12の範囲をとる。  
２が最も能力が低く12が最も能力が高い。中央値は7。
キャラクター作成時に能力は自分で決めても良いし,2d6のダイスロールでランダムに決めても良い。  
- 洞察力
  * 物事の本質を理解したり,なにかを学習する能力
- 創造力
  * 何かを作り出す能力,問題解決等で別な解決方法を見つけ出す能力
- 共感力
  * 人間関係を円滑に保つ能力
#### 能力値の成否判定時の補正値
能力値に関連した成否判定を行う場合に,能力値の数値によって補正をかけたい場合には以下を用いる
 | 能力値 | 補正値 |
 |:-----:|:------:|
 | 2-3   | -2     |
 | 4-5   | -1     |
 | 6-8   | 0      |
 | 9-10  | +1     |
 | 11-12 | +2     |


## スキル
キャラクターが保持している技能を指す。
スキル名の後ろの数値はスキルレベルと呼称する。
スキルレベルは0-5の範囲をとる。  
0が最も能力が低く,5が最も高い。
スキルの数値は成否判定の際のプラスの補正値として作用する。
スキル0は最低限の知識を持っている初心者状態を表している。
スキルが無いキャラクターが,そのスキルを必要とするタスクの成否判定を行う場合-2の補正がつく。

### スキルレベルの目安

| スキルレベル | 目安    |
|:------------:|:-------|
| なし         | 名前くらいは聞いたことがあるかもしれないが,なにもわかっていない |
| 0            | 最低限の知識を持っているが指導者がいなければタスクを遂行できない |
| 1            | 簡単なタスクを遂行できるが難解なケースでは上級者の支援を必要とする |
| 2            | 一人でタスクを完了できる |
| 3            | タスクを完了する時間を短縮できる場合がある,タスクを分解したり統合したりできる |
| 4            | プロジェクト全体のタスク完了の成功率を上げることがある |


※ 補足：スキルレベルは組み込みスキル標準のスキル基準のレベルとほぼ一致していると考えて良い。


[IPA組み込みスキル標準スキル基準] (https://www.ipa.go.jp/archive/files/000023849.pdf)


### スキルの表記
スキルは,キャラクターシートに\<スキル名 - スキル数値>の書式で記載する。  
例：\<Java言語 - 1>,\<システム設計 - 2>

### スキルの種類
  * 各種プログラミング言語（例：C,Java,Python,MATLAB,Verilogなど）
  * プラットフォーム知識(例：Linux,Windows,iOS,ROS,AUTOSARなど)
  * ハードウェア知識(例：アナログ回路知識,デジタル回路知識など)
  * ミドルウェア知識(例：Node.js,DBMSなど)
  * 開発ツール&開発環境知識（例：構成管理ツール,テストツール,エディタ,Office系アプリ,自動化ツール,LLM活用など）
  * 通信プロトコル知識(例：TCP/IP,CAN,FlexRay)
  * 設計系スキル(要求開発,プロトタイピング,システム設計,詳細設計,テスト設計など)
  * ドメイン知識(銀行業務知識,Eコマース知識,自動車開発知識など)
  * 管理系スキル(スケジュール管理,リスク管理,査定,面談など)
  * 調達系スキル(見積もり,発注など)
  * 交渉系スキル(プレゼンテーション,根回し,価格交渉,会議司会進行)

### スキルの成長（オプション）
 ゲームプレイ時に,成否判定のダイスロールの結果が12以上の場合,スキルに経験を付与する。  
 GMが定めた間隔毎に（例えば毎週最終稼働日）スキルの経験が現在のスキル数を上回っている場合にスキルレベル上昇チェックを行う。  
#### スキルレベル上昇チェック
 成功基準値 = 7 + 現在のスキルレベル
 成功基準値の上限は12とする
 2d6で成功基準値を上回った場合,チェック対象のスキルレベルを１増加させる。

### 特性
キャラクターの特性は特定の判定への修正として作用する。
以下に例をあげるが,例にある以外に自由に作成して良い。
プレイ中なんらかの印象的な出来事が発生した場合にGMが特性を追加しても良い  

特性の例
| 特性の名称 | 説明                 |
|:----------|:---------------------|
| 健康優良   | ヘルスチェック時に+1の補正,ヘルスペナルティチェック時に+1の補正 |
| 虚弱      | ヘルスチェック時に-1の補正,ヘルスペナルティチェック時に-1の補正 |
| 不屈      | メンタルヘルスチェック時に+1の補正,メンタルヘルスペナルティチェック時に +1の補正 |
| 打たれ弱い | メンタルヘルスチェック時に-1の補正,メンタルヘルスペナルティチェック時に -1の補正 |
| ムードメーカー | チームの雰囲気に関するロールに+1の補正 |
| 癒し系 | チームの雰囲気に関するロールに+1の補正 |
| 陰キャ | チームの雰囲気に関するロールに-1の補正 |
| 陽キャ | チームの雰囲気に関するロールに+1の補正 |
| 癇癪持ち | チームの雰囲気に関するロールに-1の補正 |
| 趣味人 | ゲーム開始時に、1d6/2日を「必ず休む日」に指定する、「必ず休む日」に休めないとメンタルを-2しチームの雰囲気を-1する |
| 家庭の事情 | 休日の回復行動回数が常に-1される |
| 面倒なパートナー持ち | 一日の初めに2d6し4以下がでるとその日の行動余力が-1される |
| 強運の持ち主 | 週に1回だけ、自分を含めて全てのキャラクターのロール結果に1d6の補正を加えることができる,結果を見てから使うことにしても良い |
| 不運 | 補正前のロールが3以下で大失敗になる |
　| 美形 | 複数人が同じ行動をするとき異性のキャラクターのロールに+1の補正、同性のキャラクターのロールに-1の補正 |

### ヘルスポイント
キャラクターの肉体的な健康度合いを表す。
０－５の範囲で変化する。
特に指定がない場合は３ポイントでシナリオを開始する。
ヘルスポイントは,回復行動や余力（後述）の使いすぎ,ランダムイベントなどで増減する。
０になると不健康イベントが発生する。

### メンタルヘルスポイント
キャラクターの精神的な健康度合いを表す。
変化の範囲や増減タイミングはヘルスポイントと同様。
特に指定がない場合は３ポイントでシナリオを開始する。

### 開発チーム
開発チームは,特定のプロダクトを遂行する為に集まったプレイヤーもしくはノンプレイヤーキャラクターの集団を指す。
ファンタジーRPGでいうところのパーティーに該当する。
本ゲームでは,開発チーム自体がいくつかの特性を保持している。

#### 開発チームの特性
 開発チームの特性は通常,シナリオ開始時にGMにより提示されたものを使用する。
 ##### 開発チームパラメータ
  各パラメーターは -2 から +2 の範囲をとる。  
  | パラメータ | 説明  |
  |:----------|:------|
  | 開発難易度補正 | 職場で利用できる装備の質や,職場での制約,セキュリティルールによる開発難易度を補正する |
  | チームの雰囲気 | チーム内の人間関係や,モチベーションの相対的な状況を表す。チームの雰囲気は,メンタルヘルスチェックやランダムイベントに影響を及ぼす |



##### 採用している開発プロセス
 開発チームがどのような開発プロセスを採用しているかを表す。

#### 就業規則
 開発チームが所属する組織で採用されている残業時間に関する就業規則。  
 イテレーション毎に追加作業を何回まで行うことができるかを規定する。  
 組織によっては,他にも制約がある場合がある。

##### 関連する組織
 開発チームがプロジェクトを遂行する上で関わることになる組織を表す。

## 行為判定の基本
ゲーム中,成功するか失敗するか分からない行動について判定を行うことを行為判定と呼称する。  
行為判定は常に2d6のダイスロールを行うことで判定する。  
ダイスロールがGMが定める難易度と同じか大きい場合に行為判定は成功したとみなす。  
補正がある場合はダイスロールに対して行う。  
例えば何らかのスキルを用いた行為判定の場合,ダイスロールにスキルレベルを加算する。


### 行為の難易度の目安

| 難易度  | 目安  |
|:------:|:-----:|
|    4   | とても簡単    |
|    6   | 普通         |
|    8   | 少し難しい    |
|   10   | かなり難しい  |

#### 大成功と大失敗
補正前のダイスロールが12の場合と,補正後のダイスロールの結果が12以上の場合,行為判定は大成功とみなす。逆に,補正前のダイスロールが2の場合と,補正後のダイスロールが2以下の場合,行為判定は大失敗とみなす。  
2d6の最大値(12)＋補正値の結果絶対に難易度に達しない行為であったとしても補正前ダイスロールが12の場合は成功する。
同様に2d6の最低値(2)＋補正値の結果絶対に成功する行為であったとしても補正前ダイスロールが2の場合は失敗する。  
大失敗以外では絶対に成功する行為判定の場合,ロールを行わずに行為判定を成功にすることにしても良い。ただし,その場合は大成功も発生しない。

大成功,大失敗の影響はそれぞれの判定内容に予め定められたものを用いるか,無い場合はGMがアドリブで結果を補正する。(GMの判断で無視しても良い)


## ゲームの進行
### イテレーション
本ゲームにおける開発行動を行う際の時間単位を指す。ターンと言い換えても良い。  
１回のイテレーションの実際の期間はシナリオによって異なる。

#### イテレーションの期間（長さ）による行動回数の変化

  | 期間  | 通常行動回数 | 行動余力 | 行動余力使用限界 |
  |:-----:|:-----------:|:-------:|:--------------:|
  | １日  | 2       | 1      | 2  | 
  | １週  | 3       | 2      | 2  | 
  | ２週  | 4       | 2      | 3  | 
  | １月  | 5       | 3      | 4  | 
休日行動は週休２日の場合の本来休日である日を作業に当てた場合の余力を示している。  

#### 通常行動回数
 行動回数とは,一回のイテレーションでキャラクターが開発行動を行える回数を指す。
 マネージャー役のキャラクターは行動回数のみでプロジェクトを完遂できるようにスケジュールすることが理想とされる。
#### 行動余力
 行動余力とは,通常の行動回数に追加して開発行動を行う余地を示している。
  各キャラクターは通常行動回数とは別に,行動余力と同じ回数までの開発行動をペナルティなしで行うことができる。
#### 行動余力使用限界  
  キャラクターは行動余力の回数を超えて開発行動を行うことができる。  
  ただし,行動余力使用限界を超えて開発行動を行うことはできない。  
  行動余力を超えて開発行動を行う毎に,ストレスチェックを行う(疲労によるストレスチェック)  
  ヘルスポイント,メンタルヘルスポイントの何れかが０になった場合,直ちにペナルティチェックを行う。  
  失うのがどちらのポイントであるかをGMが指定しても良い。

#### ストレスチェック  
タスクの内容にヘルス,メンタルヘルスのどちらに影響するか指示がある場合はその指示に従う。  
指示が特に無い場合はランダムに決める（1D6で１-３の場合,ヘルスポイントについてのストレスチェックを行う。4-6の場合メンタルヘルスポイントについてのストレスチェックを行う）
ストレスチェックは基本難易度を7として2d6のロールを行う。キャラクターが何らかの有利または不利な特性を持っている場合はロールに補正値を加算する。さらにチームの雰囲気を補正として加算する。ロールと補正の結果が難易度を上回っている場合はストレスチェックに成功する。失敗した場合,対象としたポイントを１減らす。  

``` 
[ストレスチェック]
基本難易度 7
ロール補正
 ・ ストレスチェックに対応したキャラクターの特性
 ・ 場の空気に対応したキャラクターの特性
 ・ チームの雰囲気
結果
　・大成功：何も起きない
　・成功：何も起きない
　・失敗：ヘルスか,メンタルヘルスどちらかを1ポイント減らす
　・大失敗：失敗と同じ,もしくはGMが追加の影響を判断する
```

  | コーディング | 言語スキル | プロダクトの完成度を進める |
  | 単体テスト | 言語スキル | プロダクトの完成度を進める |
  | コーディング＆単体テスト | 言語スキル | プロダクトの完成度を進める |

  | 機能テスト | テスト | プロダクトの品質を高める |
  | 総合テスト | テスト | プロダクトの品質を高める |

| スケジュール策定 | プロジェクト管理 | プロジェクトの初回に行うスケジューリング |
| スケジュール調整 | プロジェクト管理 | スケジュールの内容調整 |
| 外部組織との交渉 | 折衝 | チーム外の組織との調整,認識合わせ等を行う,請負の受注交渉,外部発注の受注交渉 |

#### 開発外行動

#### 回復行動

 「家庭の事情」など回復行動に影響する特性を持ったキャラクターは回復行動が減少する場合がある。
 休日の場合、通常行動回数までの回復行動を行えるが、行動余力は回復行動に使用できない。
 休日に開発行動を行った場合も、行動余力を回復行動に使用できない。
 例1：イテレーションが1日の時、2回の行動を行って帰宅した。開発余力1は使っていなかったので、ヘルス回復行動を1回行った。  
 休日の場合、一日につき通常行動と同数の回復行動を行うことができる。
 例2：イテレーションが1日の時、1回の通常行動を行って帰宅した。余力と合わせて2回の行動をヘルス回復行動、メンタルヘルス回復行動１回ずつに使用した。
 例3:祝日とあわせて３連休だったので6回の回復行動を行った。
 例4:7日間の長期連休だったので14回の回復行動を行った。
 例5:土曜日は休日出勤し2回の通常行動を行ったので回復行動は行えなかった。日曜日に2回の回復行動を行った。
 例6:土曜日は休日出勤し1回の通常行動を行った通常行動が1を回復行動に使用したかったが「家庭の事情」特性を持っていたので回復行動回数が-1されたため土曜日には回復行動を行えなかった、同様に日曜日も「家庭の事情」特性を持っていたので回復行動回数が-1された、1回の回復行動を行った。

## タスク（作業と成果物）
タスク（作業と成果物）はソフトウェア開発における一連の作業の総称です。本項では作業をこなし成果物を作成する方法を解説します。  
※タスクは他のTRPGにおける倒すべき敵のような位置づけ,作業は戦闘システムに該当します。

### タスク（作業と成果物）の基本パラメータ
すべてのタスクには以下のパラメータがあります。
#### 名称
それぞれのタスクの名前です。
#### 行動の種類
どの開発行動に該当するかを表します。
#### 成果物
開発行動の結果得られる成果物です。
#### 成果物の品質
作業開始時は０です。作業判定の成否によって増減します。  
作業の品質は値が大きいほどよい結果であるとみなします。  
#### 基本難易度
開発行動の最初の難易度です,いくつかの環境要因で変化する可能性があります。  
#### 作業量
何回の開発行動成功が必要なのかを表します。
#### 作業進捗
作業の進捗状況を表します。  
作業開始時は０です。  
作業進捗が作業量を上回れば作業完了です。
#### タスクの例
```
　タスクの名前：◯△機能の詳細設計  
　行動の種類：詳細設計  
　成果物：詳細設計書  
　成果物の品質：0  
　基本難易度：6(普通)  
　作業量:2  
  作業進捗:0  
```
```
　タスクの名前：◯△機能の実装  
　タスクの種類：コーディング＆単体テスト  
　行動の種類：コード,単体テスト結果
　成果物：0  
　基本難易度：6(普通)  
　作業量:2  
  作業進捗:0  
```
```
　タスクの名前：◯△機能の実機テスト
　行動の種類：総合テスト 
　成果物：検査成績書 
　成果物の品質：0  
　基本難易度：6(普通)  
　作業量:3 
  作業進捗:0  
```

### タスクの解決
 タスクを担当したプレイヤーは,１回の行動としてタスクの解決を実施できる。  
 プレイヤーは基本難易度＋環境要因補正の難易度を対象とした行為判定を行う
 行為判定の結果によって作業進捗を増減します。
 作業進捗が,作業量以上になれば作業は完了できる状態になったとみなします。
#### 行為判定結果と進捗の上限
行為判定の結果,作業進捗がどのように変化するかを表します。

| 行為判定結果  | 作業進捗の増減 | オプション |
|:------------:|:------------:|:----------:|
| 大失敗 | -1  | 進捗か品質のどちらかを-1 |
| 失敗   | ±0  | 進捗を＋１する代わりに品質を -1 |
| 成功   | +1  | 進捗か品質のどちらかを +1 |
| 大成功 | +2  | 進捗か品質のどちらかを+2もしくは両方を+1 |


### タスクの連続性
ソフトウェア開発において,ほとんどのタスクには連続性があります。例えば要求定義→システム設計→機能設計→コーディングデバッグといった具合です。  
こうした連続性があるタスクの場合,あるタスクの後ろの工程のタスクは前の工程の成果物が入力となります。そうした場合,後ろの工程は前の工程の成果物の品質の影響を受けることになります。  
上記の理由から本ゲームでは,タスクに連続性がある場合,成果物品質の初期値は前の工程の完了時品質を継承します。
```
プレイ時の工夫[タスクカード]
 一つのタスク毎に名刺サイズのカードにしておくことを推奨します。  
 リーダー役のプレイヤーはタスクカードを配ることで,他のプレイヤーに仕事を割当るイメージです。  
```

### タスクの分割
GMが許可した場合,作業量が多いタスクを複数のタスクへ分割することを試みることができます。  
すでに作業を開始しているタスクの作業を分割する場合,分割後のタスクの何れか１つが作業進捗を継承します。  
２分割,もしくは３分割で作業量が割り切れない場合は作業量を分割するタスクに自由に割り振って構いません。ただし,作業量の合計は元のタスクと同じである必要があります。（例外：大失敗した場合）
※注釈：分割後のタスク作業量は整数である必要があります。

``` 
[タスク分割]
基本難易度 分割対象のタスクの基本難易度 + 1
ロール補正
 ・ 洞察力(もしくは関連するスキルのどちらか)
 ・ ２分割 ± 0
 ・ 3分割 - 1
 ・ 1-3の作業量に切り分け ± 0(1d6のロール結果を２で割る)
 ・ 意図した作業量に切り分け -1
結果
　・大成功：開始時品質に+1
　・成功：意図したとおりに分割できる
　・失敗：分割できない
　・大失敗：分割後のタスクの作業量をそれぞれ+1
```
### タスク開示のタイミング
一部のタスクはシナリオ開始のタイミングではプレイヤーに開示されていないものが含まれていることがあります。  
例えば,機能設計に関するタスクは前段のタスクであるシステム設計が完了しないと開示されないといった具合です。  
どのタイミングでどのタスクが開示されるかは,シナリオの設定によります。
※補足：将来追加予定の上級ルールでは,タスクそのものをプレイヤーが作成できるようにすることも検討しています。基本ルールでは予め定められたタスクを完了させるようなプレイスタイルになります。

### 特殊な行動解決
#### レビュー
レビューを行うには参加者全員が行動を消費する必要があります。  
対象の成果物を作成した参加者以外の参加者はそれぞれ行動解決のためのロールを行います。  
難易度は、該当する成果物の作成のための難易度 + 現在の品質です。　
該当するスキルもしくは洞察力でロールを補正します。　　
成功した場合、成功した参加者一人に付き+1の品質上昇の権利を得ます。  
品質上昇の権利は対象の成果物の品質を+1する代わりに追加の作業量を1増やすことができます。  
品質上昇の権利をどのように使うかは参加者全員が話し合って決めます。  
作業量を増やした場合、対象の作業を再度完了させる必要があります。  
一度のレビューで対象とできる成果物数はイテレーションの通常行動回数と同じです。
参加者の何れかが大成功した場合は、品質上昇の権利を+2できます。
参加者の何れかが大失敗した場合は、品質上昇の権利を-1します。
品質上昇の権利の合計がマイナスになってしまった場合、マイナス分を対象成果物の品質を減らすか、作業量を増やすか何れかの方法で処理します。
例：３人が設計レビューをしています。イテレーションは１日で通常行動回数が２なので、２件の成果物に対してレビューを行うことができますが今回の対象は１件の成果物です。
対象成果物の作成者以外の２人が行動判定を行い２人共成功しました。よって品質上昇の権利+2を得ました。  
この結果を受けて、３人の話し合いで+1の品質上昇と引き換えに+1の作業量増加とすることにしました。
※補足：レビューを行う文化は、一つ一つのタスクの作業量が多い開発でよく見られます。
  逆にアジャイル開発のような、一つ一つのタスクの作業量が小さい開発の場合は別な手段（ペアプログラミングやテスティングフレームワーク）で品質を担保していることが多いです。

#### ペアプログラミング
アジャイル開発プロセスのプラクティスの一つで,２人の開発者が同時にプログラミングします。  
ペアプログラミングの行動解決は以下のうように行います。  
２人のキャラクターがそれぞれ行動を消費する。行動解決は２人とも行いどちらか良い方の結果を採用します。  
どちらかのロールがスキル経験を満たす結果になった場合,本人のロールでなくてもスキル経験が付きます。  
どちらかが大失敗の結果を出した場合、もしくは両方失敗だった場合ペナルティとして二人共ストレスチェックをしなければなりません。これはペアプログラミング不調によるストレスチェックです。  
※補足：二人共大失敗だったとしても、ペアプログラミング不調によるストレスチェックは１回です。
※補足：行動余力を超えて行動している場合、二回ストレスチェックを行うことになります。(疲労によるストレスチェック)  

「ムードメーカー」や「癇癪持ち」のような場の空気に影響する特性を持っているキャラクターが参加している場合、このときの双方のストレスチェックのロール補正に影響します。
※補足：ペアプログラミングの利点は必要な工数が２倍になる代わりにミスを防げることと、お互いのノウハウを共有できることです。欠点はペアの相性によっては精神的に疲労する可能性があることです。
``` 
[ペアプログラミング]
基本難易度 タスクの基本難易度
ロール補正
 ・ 関連するスキル
結果
　・大成功：品質もしくは作業進捗+2(どちらか片方でも良いし+1ずつ割り振っても良い)
　・成功：品質もしくは作業進捗+1(どちらか片方でも良いし+1ずつ割り振っても良い)
　・失敗：変化なし、もしくは品質を-1して進捗を+1
　・大失敗：品質もしくは作業進捗-1(どちらか片方を-1する)
```
例：A君とB君がペアプログラミングをしています。A君は \<Java-4> B君は \<Java-1> です。  
難易度6のタスクに対する行動でA君は2d6 -> 10 + 4 = 14,B君は 2d6 -> 5 + 1 = 6でした。  
どちらも成功していますが,A君は大成功なのでA君の結果を採用することにしました。  
A君は同時にスキル経験の獲得要件を満たしています。B君のロールではスキル経験の獲得要件を満たしていませんが,ペアプログラミング中なのでB君もスキル経験を得ることができます。

#### 勉強会
勉強会は、参加者が特定のスキルについて学ぶ機会を提供するものです。  
勉強会には講師が必要で、講師は勉強会に先立って準備が必要です。

```
　・大失敗：勉強会の資料の品質 -1
成果物
　・勉強会の資料
補足：勉強会の準備で大成功した場合、対象スキルのスキル経験が得られます。
```
勉強会の参加者は講師含めて１回の行動となります。

### スケジュール管理とタスクの割当

### ペナルティチェック
ヘルスポイントもしくはメンタルヘルスポイントが０になった場合、直ちにペナルティチェックを行う。
ペナルティチェックは身体もしくは精神に関する異常が発生したことを表しています。  
表の内容からも分かるように、ペナルティは当事者のみならずチームの雰囲気も悪くする。ペナルティチェックが発生すること自体を避けるのが望ましい。
```
[ヘルスペナルティチェック]
以下の表で2d6の結果を反映する。
ロール補正
 ・ ヘルスに関する特性
```

| ロール | 結果         |
|:------:|:------------|
